# Commit 1 Reflection notes
Pada bagian handle_connection, saya mencoba untuk memahami bagaimana Rust membaca request HTTP secara bertahap dan terstruktur. Penggunaan BufReader membantu proses pembacaan menjadi lebih efisien karena data tidak langsung diambil mentah dari stream, melainkan melalui buffer. Method .lines() memungkinkan setiap baris dibaca satu per satu dengan pendekatan iterator. Proses ini kemudian dihentikan saat menemui baris kosong dengan .take_while, yang menandakan akhir dari header HTTP. Dengan cara ini, parsing request bisa dilakukan tanpa perlu mengelola byte secara manual. Bagian ini menunjukkan bagaimana Rust tetap memberikan kontrol, tetapi dengan alur yang lebih rapi dan mudah diikuti.

# Commit 2 Reflection notes
Saya melihat bagaimana response HTTP mulai dibangun secara langsung dari file yang dibaca. Server menerima request, lalu mengambil isi file sebagai body response. Setelah itu, response disusun dengan format HTTP yang benar sebelum dikirim ke client. Hal yang cukup penting adalah penyusunan status line dan Content-Length agar response bisa dipahami dengan baik. Proses ini memperlihatkan hubungan antara file system dan network dalam satu alur. Sehingga konsep dasar web server dalam Rust bisa terlihat lebih jelas.

# Commit 3 Reflection notes
Pemisahan response dilakukan dengan menentukan status_line dan filename terlebih dahulu berdasarkan request. Setelah itu, proses membaca file dan menyusun HTTP response dilakukan. Pendekatan ini memisahkan bagian yang berubah dan bagian yang tetap dalam kode. Dengan begitu, struktur program menjadi lebih jelas dan tidak berulang. Refactoring ini mengurangi duplikasi kode yang sebelumnya muncul di setiap kondisi if-else. Selain itu, kode juga menjadi lebih mudah dikembangkan ketika ingin menambah route baru.

# Commit 4 Reflection notes
Penggunaan match membuat penanganan request menjadi lebih terstruktur dibandingkan if-else. Setiap pola request langsung dipasangkan dengan response yang sesuai. Hal ini membuat alur routing menjadi lebih jelas dan mudah dibaca. Penambahan endpoint seperti /sleep menunjukkan adanya simulasi request yang membutuhkan waktu lama. Dari sini terlihat bahwa server masih memproses request secara berurutan. Kondisi ini membantu memahami keterbatasan pendekatan single-threaded.

# Commit 5 Reflection notes
ThreadPool digunakan untuk menangani banyak request tanpa harus membuat thread baru setiap kali ada koneksi masuk. Setiap tugas dibungkus sebagai job dan dikirim melalui channel ke worker yang sudah tersedia. Worker kemudian mengambil job tersebut dan mengeksekusinya secara paralel. Pendekatan ini membuat penggunaan resource jadi lebih efisien karena jumlah thread sudah dibatasi sejak awal. Selain itu, mekanisme ini juga menghindari masalah ketika terlalu banyak thread dibuat dalam waktu bersamaan. Dengan adanya ThreadPool, server tidak lagi terblokir oleh satu request yang lama, sehingga performanya menjadi lebih stabil saat menangani banyak request.

# Bonus Reflection notes
Penggunaan fungsi build memberikan cara inisialisasi yang lebih aman dibandingkan new. Jika sebelumnya new langsung menghentikan program saat input tidak valid, build memungkinkan error dikembalikan dan ditangani dengan lebih fleksibel. Hal ini membuat program tidak langsung panic dan lebih siap untuk kondisi yang tidak diharapkan. Selain itu, perbedaan ini juga menunjukkan bahwa cara membuat objek bisa disesuaikan dengan kebutuhan, terutama jika ada kemungkinan gagal saat inisialisasi. Pendekatan ini membantu menjaga stabilitas program ketika menghadapi input yang tidak valid. Dengan demikian, alur program menjadi lebih terkontrol dan mudah dikembangkan ke depannya.
